<summary>Snowflake Clustering Keys</summary>
  
### Overview : Snowflake automatically clusters data in micro‑partitions, but as tables grow and undergo heavy DML, natural clustering can degrade.  
**Clustering keys** allow Snowflake to maintain optimal micro‑partition organization for faster pruning and improved query performance.

# 1. What Are Clustering Keys?

A **clustering key** is one or more columns (or expressions) that define how Snowflake should organize micro‑partitions for a table or materialized view.

### Key Points
- Improve **pruning efficiency** → fewer micro‑partitions scanned.
- Improve **query performance** on large tables.
- Snowflake automatically collects clustering metadata.
- Clustering keys do **not** physically sort data; they guide micro‑partition organization.

# 2. Why Clustering Matters

Snowflake stores data in **micro‑partitions** (50–500 MB).  
Each micro‑partition contains metadata such as:
- Min/max values per column  
- Distinct values  
- Null counts  

When queries filter on columns with good clustering:
- Snowflake prunes unnecessary micro‑partitions.
- Query performance improves significantly.

When clustering degrades:
- More micro‑partitions overlap.
- Pruning becomes less effective.
- Queries scan more data → slower & more expensive.

# 3. When to Use Clustering Keys

Clustering keys are recommended when:
- Tables are **very large** (hundreds of GBs to TBs).
- Queries frequently filter on the same columns.
- Natural clustering degrades due to:
  - Frequent inserts
  - Updates
  - Deletes
  - Unsorted data ingestion

Common clustering columns:
- Dates / timestamps  
- Geographic attributes  
- IDs with natural ordering  
- Composite keys (e.g., `(country, city)`)

# 4. Defining Clustering Keys

### Create a table with a clustering key
```sql
CREATE TABLE sales (
  id INT,
  region STRING,
  sale_date DATE
)
CLUSTER BY (sale_date);

```

Add a clustering key to an existing table
ALTER TABLE sales CLUSTER BY (region, sale_date);
Remove clustering key
ALTER TABLE sales DROP CLUSTERING KEY;

5. Automatic Clustering
Snowflake can automatically maintain clustering using Automatic Clustering (Enterprise Edition+).
Benefits
No manual reclustering required.
Snowflake continuously monitors clustering depth.
Automatically reorganizes micro‑partitions in the background.
Without Automatic Clustering
You must manually recluster:
sql
ALTER TABLE sales RECLUSTER;

6. Clustering Depth
Clustering depth measures how well micro‑partitions are organized for the clustering key.
Interpretation
Depth = 1 → excellent clustering
Depth > 1 → overlapping partitions → degraded clustering
Depth = 0 → empty table
Check clustering depth
SELECT SYSTEM$CLUSTERING_DEPTH('SALES');
Check detailed clustering info
SELECT SYSTEM$CLUSTERING_INFORMATION('SALES');
# Snowflake Clustering — Cost Considerations & Best Practices  
### Concise Summary (Markdown Version)

## 7. Cost Considerations

Clustering affects **compute**, not storage.

### 🔹 Costs Come From
- Reclustering operations (manual or automatic)  
- Large tables with frequent DML (INSERT/UPDATE/DELETE)  
- Multi‑column clustering keys (more expensive to maintain)  

### 🔹 Best Practices
- Use clustering keys **only on large tables**.  
- Avoid clustering **small or medium** tables.  
- Choose clustering columns with:
  - **High cardinality**  
  - **Frequent filtering**  
  - **Natural ordering** (e.g., timestamps)  

## 8. Best Practices for Choosing Clustering Keys

### ✔ Good Candidates
- Timestamp columns used in **range queries**  
- Geographic or categorical columns with **high cardinality**  
- Composite keys used in **WHERE** clauses  

### ❌ Avoid
- Low‑cardinality columns (e.g., TRUE/FALSE)  
- Columns rarely used in filters  
- Columns with **random distribution** (UUIDs)  

### 🔹 General Guidelines
- Start with **one** clustering key.  
- Add more **only if necessary**.  
- Monitor clustering depth regularly.

# Snowflake Partition Pruning (COF-C03 Exam Notes)

## 📘 Micro-partitions (Main Concept)
https://docs.snowflake.com/en/user-guide/tables-clustering-micropartitions

# 🧠 What is Partition Pruning?

Partition pruning is a performance optimization technique where:
👉 Snowflake skips scanning unnecessary data

## 📌 How It Works

Snowflake stores metadata for each micro-partition:
- Minimum value
- Maximum value
- Number of distinct values

## 📌 Example

```sql
SELECT * FROM orders
WHERE order_date = '2024-01-01';
```

## 👉 Snowflake will:

- Check metadata of each micro-partition  
- Skip partitions where date does not match  
- Scan only relevant data  

## ⚙️ Why Partition Pruning Works

Because of:

- Micro-partitions  
- Metadata (min/max values)  
- Columnar storage  

## 🚀 Benefits

- Faster query execution  
- Reduced data scan  
- Lower compute cost  

## 🔥 Important Exam Points (COF-C03)

### ✅ Partition pruning is:

- Automatic  
- Managed by Snowflake  
- Based on metadata  

### ❌ NOT:

- Manual partitioning  
- User-defined partitions (like Hive/Spark)  

## ⚠️ Common Mistake

- ❌ Thinking Snowflake uses manual partitions  
- ❌ Confusing with Spark/Hive partitioning  

## 🔗 Related Topics to Study

- Micro-partitions  
- Clustering keys  
- Query performance optimization  

## 🎯 One-Line Summary
👉 Snowflake uses micro-partition metadata to skip unnecessary data during query execution.

<summary>Snowflake Spark Connector</summary>

## 📌 Overview
The **Snowflake Connector for Spark** enables Apache Spark to read and write data directly to Snowflake, treating Snowflake as a **data source and sink**. It supports **bi‑directional data movement**, pushdown optimization, and secure integration with Spark clusters.


# 1. What the Connector Does

### 🔹 Core Capabilities
- Read Snowflake tables into Spark DataFrames  
- Write Spark DataFrames into Snowflake tables  
- Push down SQL operations to Snowflake for performance  
- Support for structured, semi‑structured, and large datasets  
- Secure authentication (password, OAuth, key‑pair)

### 🔹 When to Use Spark Connector
Use the Spark Connector when:
- You already run Spark workloads  
- You need to process external data in Spark before loading to Snowflake  
- You want to join Snowflake data with Spark‑resident data  
- You need to enforce **row/column policies** on Iceberg tables from Spark  

### 🔹 When to Prefer Snowpark Instead
Snowpark is recommended when:
- You want all compute to run inside Snowflake  
- You want full pushdown of logic (including UDFs)  
- You want to avoid managing Spark clusters  

# 2. Architecture & Interaction

### 🔹 How Spark & Snowflake Communicate
- Spark uses the connector + Snowflake JDBC driver  
- Data is exchanged via **staging** (internal or external)  
- SQL operations are pushed down to Snowflake when possible  
- Spark executes only operations that cannot be pushed down  

### 🔹 Bi‑Directional Data Flow
| Direction | Description |
|----------|-------------|
| **Spark → Snowflake** | Writes DataFrames to Snowflake tables |
| **Snowflake → Spark** | Reads Snowflake tables into Spark DataFrames |


# 3. Key Features

### 🔹 Pushdown Optimization
The connector pushes operations such as:
- Filters  
- Projections  
- Aggregations  
- Joins (when possible)  

This reduces data movement and improves performance.

### 🔹 Supported File Formats for Staging
- Parquet  
- CSV  
- JSON  

### 🔹 Supported Spark Languages
- Scala  
- Python (PySpark)  
- Java  

# 4. Configuration

### 🔹 Required Options
Typical connection options include:
- `sfURL`  
- `sfUser`  
- `sfPassword` or key‑pair  
- `sfDatabase`  
- `sfSchema`  
- `sfWarehouse`  
- `sfRole` (optional)

### 🔹 Example (Scala)
```scala
val options = Map(
  "sfURL" -> "account.snowflakecomputing.com",
  "sfUser" -> "USER",
  "sfPassword" -> "PASSWORD",
  "sfDatabase" -> "MYDB",
  "sfSchema" -> "PUBLIC",
  "sfWarehouse" -> "COMPUTE_WH"
)
```

spark.read
  .format("snowflake")
  .options(options)
  .option("dbtable", "MY_TABLE")
  .load()

  df.write
  .format("snowflake")
  .options(options)
  .option("dbtable", "MY_TABLE")
  .mode("overwrite")
  .save()

**6. Authentication Methods**
Username + password
Key‑pair authentication
OAuth

External browser authentication (for interactive use)

**7. Performance Considerations**
🔹 Best Practices
Enable pushdown whenever possible
Use appropriate warehouse size for large reads/writes
Use Parquet staging for large data transfers
Avoid unnecessary repartitioning in Spark
Use Snowflake for heavy SQL operations instead of Spark

**🔹 Limitations**
Some Spark operations cannot be pushed down
Complex UDFs run on Spark, not Snowflake
Large writes may require staging area tuning

**8. Use Cases**
ETL/ELT pipelines combining Spark + Snowflake
Machine learning workflows using Spark MLlib
Processing large external datasets before loading to Snowflake
Querying Iceberg tables with Snowflake policies enforced

9. Key Takeaways
The Snowflake Spark Connector enables fast, secure, bi‑directional data movement.
Pushdown optimization minimizes data transfer and improves performance.
Snowpark is preferred when you want all compute inside Snowflake.
Spark Connector is ideal when you must run Spark‑native workloads or integrate external data.
